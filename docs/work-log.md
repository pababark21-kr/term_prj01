# 종합 작업 로그 - 냉장고 파먹기 / fridge-chef

## 1. 최종 제출 대상

최종 제출 대상은 냉장고 파먹기 / fridge-chef입니다. 이 서비스는 1인 가구 자취생이 냉장고 속 보유 재료를 입력하면 만들 수 있는 레시피를 추천하고, 부족한 재료를 장보기 행동으로 연결하는 모바일 UI/UX 디자인 시안입니다.

최종 화면은 메인 화면, 레시피 목록 화면, 레시피 상세 화면 3종으로 구성됩니다.

## 2. 사용자 목표와 핵심 플로우

사용자 목표는 이미 가진 재료로 바로 만들 수 있는 요리를 빠르게 찾고, 부족한 재료가 있을 때 다음 행동으로 자연스럽게 이동하는 것입니다.

핵심 플로우는 다음 3단계입니다.

1. 재료 입력: 사용자가 냉장고에 있는 재료를 태그 형태로 입력합니다.
2. 추천 레시피 비교: 입력한 재료를 기준으로 추천 레시피를 카드 목록에서 비교합니다.
3. 상세 조리/부족 재료 확인: 선택한 레시피의 조리 단계와 보유/부족 재료를 확인하고 장보기 행동으로 이어집니다.

## 3. 화면 3종의 역할

### 3.1 메인 화면

메인 화면은 재료 입력을 가장 앞에 둔 시작 화면입니다. 초기 프롬프트에서는 검색바와 재료 입력 영역이 함께 있었지만, 두 요소의 역할이 중복될 수 있어 수정 과정에서 재료 칩과 추가 버튼 중심의 입력 구조를 기록했습니다.

최종 산출물 경로: [../assets/final-ui/01-main-screen.jpg](../assets/final-ui/01-main-screen.jpg)

### 3.2 목록 화면

목록 화면은 입력 재료를 바탕으로 추천 레시피를 비교하는 화면입니다. 수정 과정에서 카드 수를 정확히 3개로 고정하고, 필터 칩, 조리 시간, 난이도, 재료 일치율, 보유 재료 근거 문구를 함께 제시하도록 정리했습니다.

최종 산출물 경로: [../assets/final-ui/02-list-screen.jpg](../assets/final-ui/02-list-screen.jpg)

### 3.3 상세 화면

상세 화면은 선택한 레시피를 실제 조리 행동으로 연결하는 화면입니다. 완성 요리 이미지, 레시피명, 조리 시간, 난이도, 1인분/칼로리 정보, 재료 칩, 부족 재료 표시, 조리 단계, 시작 버튼을 포함하도록 설계했습니다. 부족 재료에는 장보기 연결 행동을 추가해 서비스 핵심 가치를 강화했습니다.

최종 산출물 경로: [../assets/final-ui/03-detail-screen.jpg](../assets/final-ui/03-detail-screen.jpg)

## 4. 정보 구조와 우선순위

정보 구조는 사용자의 의사결정 순서를 기준으로 정리했습니다.

- 메인 화면: 재료 입력 영역과 Find Recipes 버튼을 우선 배치해 첫 행동을 분명히 합니다.
- 목록 화면: 추천 레시피 카드, 조리 시간, 난이도, 재료 일치율, 보유 재료 근거를 함께 보여 비교 기준을 제공합니다.
- 상세 화면: 완성 이미지와 기본 정보를 먼저 보여주고, 재료/부족 재료/조리 단계를 이어 배치해 실행 가능성을 확인하게 합니다.

화이트 배경과 소프트 그린 포인트 컬러는 음식/생활 앱의 가벼운 분위기와 CTA 가시성을 함께 확보하기 위한 선택입니다.

## 5. 프롬프트 개선 과정

프롬프트 로그에는 홈, 목록, 상세 화면별 초안과 수정1 단계가 기록되어 있습니다.

- 홈 화면: 검색바와 재료 입력의 역할 중복을 줄이고, 재료 칩과 추가 버튼을 중심으로 정리했습니다.
- 목록 화면: 2~3개처럼 모호했던 카드 수를 정확히 3개로 고정하고, 필터 칩과 재료 일치율 근거 문구를 추가했습니다.
- 상세 화면: 부족 재료 표시만 있던 구조에 Add to shopping list 링크, 1인분, 칼로리 정보를 추가해 장보기 흐름과 연결했습니다.

정확한 프롬프트 원문은 [prompt-log.md](prompt-log.md)와 [fridge-chef-work-log.md](fridge-chef-work-log.md)에 기록되어 있습니다.

## 6. 후가공 방식과 선택 이유

후가공은 Figma에서 최종 화면을 배치하고, 투명 Hotspot으로 화면 전환 흐름을 연결하는 방식으로 정리했습니다. repo 내부 문서에는 프로토타입 링크와 화면 연결 기록이 있으며, 외부 Figma 내용을 직접 열람했다고 가정하지 않습니다.

후가공을 선택한 이유는 이미지 생성 결과만으로는 실제 사용자 흐름을 확인하기 어렵기 때문입니다. 화면을 Figma에 배치하면 메인 -> 목록 -> 상세 흐름, CTA 위치, 화면 간 이동을 평가자가 빠르게 확인할 수 있습니다.

## 7. AI 생성 이미지의 한계와 대응

AI 생성 UI 이미지는 다음 한계가 발생할 수 있습니다.

- 한국어 텍스트 깨짐 또는 어색한 번역
- 화면별 색상/폰트/여백 일관성 차이
- 검색바와 재료 입력처럼 역할이 중복되는 UI 요소
- 부족 재료 표시가 실제 행동으로 연결되지 않는 문제

이에 대해 한국어 버전 재생성 프롬프트, HEX 색상 명시, consistent sans-serif typography, iOS status bar와 safe area, 카드 수 고정, 장보기 링크 추가 등으로 재발 방지 전략을 세웠습니다.

## 8. 일관성 유지 전략

일관성 유지를 위해 다음 기준을 반복 적용했습니다.

- 화면비: 모바일 9:16 비율 유지
- 색상: 화이트 배경과 소프트 그린 포인트 컬러 사용
- 타이포그래피: 일관된 sans-serif 스타일 유지
- 컴포넌트: 둥근 카드, 태그 칩, 주요 CTA 버튼 반복 사용
- 여백: 미니멀 UI에 맞게 넉넉한 여백 유지
- 흐름: 메인 -> 목록 -> 상세의 3단계 구조 유지

이 전략은 화면별 역할은 분명히 나누면서도 같은 서비스 안의 화면처럼 보이게 하기 위한 기준입니다.

## 9. 적용 전후 개선 요약

| 항목 | 적용 전 | 적용 후 |
| --- | --- | --- |
| 색상 | soft green처럼 추상적으로 작성 | `#A8D5BA`, `#333333`, `#FFFFFF` 등 구체 색상 기록 |
| 화면 안전영역 | 별도 명시 없음 | status bar와 safe area 명시 |
| 홈 입력 | 검색바와 재료 입력이 중복 | 재료 칩과 추가 버튼 중심으로 정리 |
| 목록 카드 | 2~3개로 모호 | 정확히 3개 카드와 필터 칩 추가 |
| 재료 일치율 | 숫자 중심 | 보유 재료 근거 문구 추가 |
| 상세 장보기 | 부족 재료 표시 중심 | Add to shopping list 링크로 행동 연결 |
| 일관성 | 화면별 스타일 차이 가능 | 동일 톤, 폰트, 컬러, 카드/버튼 규칙 반복 |
